# 1. 产品一句话定义

本系统是一个面向城市管理场景的 **社会风险主线建模与世界线推演平台**。

系统通过接入城市线上与可获得的多源数据，识别显性问题与隐性线头，将分散的数据、信号、小问题、诱因和支点建模为可解释的候选主线；在人工确认后生成 World State 输入，进行世界线推演；并通过 Agent Council 多方主体研判，评估不同主体反应对支点和分支概率的影响；在现实事件发生时，系统支持事中快响、全网快搜、实时态势拼装和临时处置建议；推演或处置完成后，系统自动进行数据打标、案例沉淀和复盘校准。

一句话：

> 把海量城市数据变成可推演的风险主线，把推演结果变成可执行建议，再把现实结果沉淀成系统能力。

# 2. 产品愿景

## 2.1 目标愿景

让城市管理者和风险研判人员能够：

1. 及时发现线上和公开数据中的潜在问题线头。
2. 识别明显问题与不起眼小问题共同指向的社会风险。
3. 理解一条风险主线是如何由数据、信号、诱因和支点形成的。
4. 推演该主线在未来可能如何变化。
5. 通过多方 Agent 模拟各主体立场、约束、误读和反应。
6. 在现实事件发生时，快速搜罗全网数据，进入事中快响。
7. 将推演结果转化为处置建议、报告和监测任务。
8. 将每次事件沉淀为案例库、前兆库、偏差库和 Agent 校准库。

# 3. 产品边界

## 3.1 系统做什么

系统负责：

1. 多源数据接入与治理。
2. 线上数据实时或准实时检索。
3. 数据源可信度、字段完整度、异常率、标签覆盖率管理。
4. 自动打标、风险语义识别、情绪识别、支点识别。
5. 显性问题识别。
6. 隐性线头聚合。
7. 系统成线建议。
8. 用户主动检索与手动主线草稿构建。
9. 主线建模与人工确认。
10. World State 输入包生成。
11. 世界线推演。
12. 外部信息与假设注入。
13. Agent Council 多主体研判。
14. 事中快响与全网快搜。
15. 处理建议与研判报告。
16. 处置任务跟踪。
17. 后台自动数据打标、案例沉淀、现实结果跟踪和复盘校准。

## 3.2 系统不做什么

系统不做：

1. 不替代城市管理者做最终决策。
2. 不基于线上热度直接判定线下事实。
3. 不对具体个人进行画像、追踪或处置建议。
4. 不输出压制性、诱导性、操控性建议。
5. 不绕过平台权限、反爬限制或数据授权边界抓取数据。
6. 不把 Agent 输出视为真实民意。
7. 不自动发布官方回应。
8. 不直接指挥线下执法或强制处置。
9. 不承诺全网所有信息绝对实时获取。
10. 不将未经人工确认的高风险结论直接进入正式报告。

# 4. 核心业务流程总览

完整业务链路如下：

```text
数据源管理
→ 数据 / 信号检索
→ 问题线索识别
→ 主线建模
→ World State 输入确认
→ 世界线推演
→ Agent Council 多方研判
→ 汇报与处置闭环
→ 现实结果跟踪
→ 案例库沉淀
```

同时系统需要支持一条并行链路：

```text
现实事件发生
→ 事中快响
→ 全网快搜
→ 实时态势拼装
→ 临时主线 / 已有主线匹配
→ 处置建议
→ Agent Council 回应压力测试
→ 处置后监测
→ 闭环复盘
```

# 5. 系统能力架构

## 5.1 主能力模块

| 模块 | 作用 |
|---|---|
| 城市驾驶舱 | 城市总体态势、候选风险方向、事中快响入口 |
| 数据源管理 Data Source Hub | 管理数据源、质量、可信度、标签体系 |
| 数据 / 信号检索 Signal Retrieval | 搜索、筛选、查相似、加入主线草稿 |
| 问题线索识别 Issue Discovery | 系统自动识别显性问题和隐性线头，提出成线建议 |
| 主线建模 Mainline Builder | 人机协同组织证据、诱因、支点，确认主线 |
| World State 输入确认 | 检查主线是否具备推演条件 |
| 世界线推演 Worldline | 生成未来可能路径，支持假设注入 |
| Agent Council | 多主体反应研判，影响支点和分支概率 |
| 事中快响 Live Response | 现实事件发生后的全网快搜、态势拼装、处置建议 |
| 现场态势增强 Field Intelligence | 接入视频、无人机、图片，补充现场事实 |
| 汇报与闭环 | 生成报告、处置建议、任务跟踪、监测重点 |
| 案例库 / 复盘中心 | 自动沉淀案例、前兆模式、偏差数据和 Agent 校准结果 |

## 8.5 页面五：主线建模工作台 Mainline Builder

### 页面主题

把系统成线建议或人工草稿组织成可解释、可确认、可进入推演的主线。

### 业务流节点

主线建模与确认。

### 页面结构

```text
顶部：主线建模状态指标
左侧：问题线索识别 + 候选主线池
中间：主线结构画布
右侧：主线解释与确认
底部：建模质量控制
```

### 主线结构画布

必须表达：

```text
起点 / 背景
→ 异常信号
→ 小问题聚合
→ 核心诱因
→ 关键支点
→ 证据缺口
```

# 9. 事中快响详细设计

## 9.1 为什么需要事中快响

事前推演回答：

```text
这件事可能会怎样？
```

事中快响回答：

```text
现在已经发生了，网上正在怎么发酵，应该先做什么？
```

## 9.2 事中快响业务流程

```text
事件触发
→ 快速建档
→ 全网快搜
→ 数据聚类与去重
→ 事实 / 情绪 / 传播三分离
→ 态势卡生成
→ 信息缺口识别
→ 处置建议
→ Agent Council 回应测试
→ 后续监测
```

## 9.3 全网快搜能力

用户输入：

```text

对象

```

系统检索：

```text
搜索热度
```

## 9.5 事实 / 情绪 / 传播三分离

# 21. 风险与应对

| 风险 | 应对 |
|---|---|
| 线上热度误判为现实风险 | 事实/情绪/传播三分离，人工确认 |
| 数据源不可用 | 数据源健康监控、降级策略、人工上传 |
| Agent 胡编 | 证据边界、输出校验、人工复核 |
| 系统过度自动判断 | 系统建议 + 人工确认 |
| 页面功能混乱 | 明确页面主对象和输出对象 |
| 实时性不足 | 流式架构 + 数据源分级 |
| 隐私风险 | 脱敏、权限、审计、最小化采集 |
| 案例库质量差 | 复盘校准、专家审核、标签规范 |
| 推演不可信 | 解释链、证据链、历史案例和偏差评估 |

# 22. 最终产品原则

1. **数据要可信**：数据源、字段、标签、可信度必须可见。
2. **线索要可找**：用户必须能主动搜索、筛选、查相似。
3. **主线要可解释**：主线必须从数据、信号、小问题、诱因、支点逐层解释。
4. **推演要可追溯**：每个分支变化都要能追溯到条件、支点和证据。
5. **Agent 要可约束**：Agent 基于证据、角色、利益和边界发言。
6. **事中要能快响**：现实发生时，快速全网检索、拼装态势、给出建议。
7. **处置要可落地**：建议必须变成报告、任务、监测点。
8. **沉淀要自动化**：数据打标、案例沉淀和复盘校准在后台自动进行。
9. **人要掌控关键判断**：系统建议，人确认。
10. **系统越用越强**：每次结果都进入案例库、前兆库、偏差库和 Agent 校准库。

# 23. 结语

本系统不是传统舆情看板，也不是单纯大模型分析工具。它的核心是建立一套城市风险认知链路：

```text
数据可信
→ 线索可找
→ 主线可建
→ 世界可推演
→ 主体可研判
→ 建议可执行
→ 结果可沉淀
```

最终目标是让城市管理从“事后反应”向“事前识别、事中快响、事后复盘”的闭环治理能力演进。

# 25. 补充需求：事前推演与事中快响双轨机制

## 25.1 需求背景

系统不能只服务“事前推演”。在真实业务中，经常存在另一类场景：现实事件已经发生，但系统之前没有识别出来；现实事件正在快速扩散，用户需要立即搜罗全网数据；某条事前世界线中的分支已经在现实中出现，需要切换到事中处置。

因此系统需要形成双轨机制：

```text
事前推演轨：发现线头 → 主线建模 → 世界线推演 → Agent Council → 预案与报告
事中快响轨：现实发生 → 全网快搜 → 实时态势拼装 → 临时处置 → 监测反馈
```

## 25.2 两种模式定义

### 事前推演模式

目标：在事件尚未明显发生前，通过数据建模和推演预判未来变化。

核心问题：这件事会不会发生、它可能怎么演化、现在做什么可以降低风险。

### 事中快响模式

目标：当现实事件已发生或正在扩散时，快速获取实时线上数据，拼装当前态势，支持临时处置。

核心问题：现在发生了什么、全网有哪些相关信息、传播到哪一步、事实/情绪/传播分别是什么状态、现在应该优先核实和回应什么。

## 25.3 触发方式

| 触发方式 | 说明 | 示例 |
|---|---|---|
| 系统告警触发 | 平台监测到异常升温或跨平台共振 | 某关键词 30 分钟增长 300% |
| 人工触发 | 用户知道现实中已发生，需要立刻搜罗全网数据 | 用户输入事件关键词、地点、对象 |
| 世界线转入 | 某个推演分支在现实中出现 | 标记某世界线节点为“现实发生” |

## 25.4 状态设计

```text
monitoring          观察中
simulating          事前推演中
live_detected       疑似现实发生
live_response       事中快响中
response_issued     已发布回应 / 已采取处置
monitoring_after    处置后监测中
closed_loop         已闭环
reviewed            已复盘
```

## 25.5 与世界线的关系

```text
实时态势 → 匹配既有主线 / 世界线节点
实时态势 → 标记某分支是否正在发生
实时态势 → 修正当前世界状态
实时态势 → 触发新一轮推演或事中处置
```

输出示例：

```text
当前实时态势与 C 线“二次升温”分支相似度：0.78
建议：切换到事中快响，并重点监测 C 线关键指标。
```

# 27. 补充需求：同城传播与圈层分析

## 27.1 业务定位

同城传播与圈层分析用于判断一个本地话题是否正在从小圈层扩散到更大的公共议题。

该模块不做个人画像，不识别个人身份，不输出针对个人的判断。系统分析的是传播簇、圈层、平台、话题和内容流转。

一句话定位：判断同城话题在不同传播圈层中的扩散结构、破圈风险和下一步传播方向。

## 27.2 需要回答的问题

```text
这个话题是否只在小圈层里发酵？
是否已经进入短视频围观圈、微博公共讨论圈、媒体圈？
哪些圈层之间发生了内容流转？
是否出现桥接节点？
话题语义是否正在从具体诉求变成公共议题？
是否需要进入事中快响？
```

## 27.3 同城态势指标

```text
城市热度
本地讨论占比
平台覆盖数
同城传播速度
区域热度分布
主要区域
主要平台
事实可信度
数据缺口
```

## 27.4 圈层类型

```text
本地居民讨论圈
社区 / 论坛讨论圈
短视频围观圈
本地自媒体圈
微博公共讨论圈
新闻媒体圈
官方回应圈
行业/专业讨论圈
```

每个圈层展示：内容量、升温速度、情绪倾向、主要关键词、代表内容、可信度、是否连接其他圈层。

## 27.5 发帖人关联的合规表达

系统不展示个人画像，而是展示脱敏传播关系：

```text
传播簇
账号簇
相似文案组
同素材传播组
同时间窗口发布组
跨平台搬运链
桥接节点
```

可分析的关系类型：

```text
转发 / 引用关系
评论互动关系
相似文本关系
相似素材关系
时间同步关系
地域共现关系
话题共现关系
跨平台桥接关系
```

禁止输出：

```text
具体个人身份
线下位置
个人画像
针对个人的处置建议
非公开个人关系推断
```

## 27.6 跨圈传播风险评分

新增指标：`Cross-Circle Spread Risk`

评分因子：

```text
圈层数量
圈层异质性
桥接节点数量
语义变形程度
传播速度
视觉素材支撑
情绪强度
官方回应缺口
跨平台共振
历史相似传播路径
```

示例输出：

```text
跨圈传播风险：高 0.76 / 1.00
原因：
1. 话题已从社区论坛扩散到短视频和微博。
2. 本地自媒体开始参与解读。
3. 语义从“补偿标准”转向“信息不透明”。
4. 暂无明确官方解释。
```

## 27.7 页面设计

页面名称：同城传播与圈层分析

页面结构：

```text
左侧：同城话题列表
中间：传播圈层图谱
右侧：跨圈传播风险解释
```

### 中间：传播圈层图谱


# 29. 补充需求：数据源实时性能与可靠性保障

## 29.1 业务定位

事中快响依赖数据时效，但系统不能承诺全网绝对实时。系统可承诺的是：

> 数据一旦进入系统，系统在规定时延内完成清洗、打标、聚合、告警和展示。

外部平台数据获取时效受接口、授权、频控、源站更新、网络和平台策略影响。

## 29.2 数据源分级

### A 类：实时源

用于事中快响：短视频热榜/同城内容、公开社媒话题、新闻快讯、论坛高频更新、搜索热度、人工上报、系统内部告警。

目标：1-5 分钟级采集，30 秒-2 分钟处理。

### B 类：准实时源

用于主线建模和趋势判断：政务公告、公众号文章、项目公示、新闻转载、公开评论、行业数据、地图/POI 信息。

目标：10-30 分钟级采集，3-10 分钟处理。

### C 类：离线源

用于案例库和历史相似：历史案例、复盘报告、政策文件、项目材料、人工上传文档。

目标：小时级 / 天级处理。

## 29.3 实时链路架构

```text
数据源 Connector
→ 采集调度器 Scheduler
→ 消息队列 Kafka / RabbitMQ
→ 流式处理 Pipeline
→ 数据清洗 / 去重 / 标签 / 情绪 / 风险语义
→ 实时索引 OpenSearch + Vector DB
→ 信号雷达 Signal Radar
→ 告警中心 Alert Center
→ Live Incident 事中事件对象
→ 事中快响工作台 Live Response
```

## 29.4 实时性能目标

| 新数据采集入库 | 1-5 分钟 |

| 信号识别 | < 3 分钟 |
| 相关数据推荐 | < 5 秒 |
| 事中快响事件生成 | < 5 分钟 |
| 实时数据流刷新 | 10-30 秒 |
| 告警推送 | < 1 分钟 |

## 29.5 降级策略

# 30. 补充需求：线上数据评分体系

## 30.1 设计目的

线上数据不能直接等同现实事实。系统必须拆分三个分数：

```text
线上热度
事实可信度
成线风险
```

## 30.2 线上热度评分

回答：这个话题在线上有多热？

因子：讨论量、互动量、传播速度、平台覆盖、搜索变化、高影响账号参与、同城热度。

## 30.3 事实可信度评分

回答：这个线上信息有多大概率对应真实问题？

因子：来源可信度、证据完整度、官方印证、多源一致性、原文/原图/原视频、是否存在反向证据、时间地点明确度、是否疑似搬运或旧内容。

## 30.4 成线风险评分

回答：这些线上信号是否可能形成风险主线？

因子：共同主题、共同区域、共同对象、共同人群、共同支点、历史前兆相似、时间窗口内升温、跨平台共振、信息缺口强度。

## 30.5 页面展示示例

| 话题 | 线上热度 | 事实可信度 | 成线风险 | 建议 |
|---|---:|---:|---:|---|
| 某短视频爆火但信息模糊 | 高 | 低 | 中 | 需核验，不宜直接推演 |
| 多个社区反复询问补偿标准 | 中 | 中高 | 高 | 建议进入主线建模 |
| 官方公告发布但评论质疑多 | 中 | 高 | 中高 | 建议 Agent Council 测试回应风险 |
| 论坛少量历史投诉复燃 | 低 | 中 | 中高 | 加入观察，查相似数据 |

# 31. 补充需求：全局流程指引设计

## 31.1 问题背景

当前原型中，部分页面底部出现流程指引，但其他页面没有，会导致用户误以为该流程是某个页面的局部组件，而不是系统级业务流程。

流程指引的价值在于提供全局方向感：当前在哪一步、这一步目标是什么、下一步是什么、能否回到上一步。

## 31.2 设计原则

流程指引不应作为单个页面的装饰组件，而应成为跨页面一致的全局导航辅助。

建议采用：顶部二级流程条 / 顶部轻量步骤条 / 页面右上角微型阶段提示 / 局部页面最小操作提示。

## 31.3 推荐流程

```text
1 数据接入与治理
2 数据检索与抓取
3 问题线索识别
4 主线建模确认
5 世界线推演
6 多方主体研判
7 汇报与闭环
```

## 31.4 页面规则

1. 所有核心流程页面都应显示当前位置。
2. 当前步骤高亮，已完成步骤可点击回看。
3. 待开始步骤只展示，不强制点击。
4. 子页面不应在底部单独新增一套不一致流程条。
5. 页面内如果空间不足，只展示“当前阶段 + 下一步动作”。

示例：

```text
当前阶段：数据检索与抓取
下一步：进入主线建模
```

```text
线上数据：判断网上怎么发酵
无人机/视频/3D：判断现场到底是什么状态
```

## 34.2 适用场景

```text
现场是否存在聚集
道路是否阻塞
出入口是否拥堵
是否存在施工围挡、车辆、障碍物
处置资源是否到场
人流是否向敏感区域移动
是否需要设置沟通点或引导区
```

## 34.3 建设层级

### L1：现场材料接入

人工上传视频/图片、直播/视频链接分析、关键帧抽取、OCR/ASR、画面摘要、隐私脱敏、现场事实卡。

### L2：授权视频 / 无人机接入

固定摄像头、无人机回传视频、授权飞行任务、视频流实时分析、现场态势地图。

### L3：无人机任务管理与轻量 3D 场景

飞行任务计划、航线管理、空域校验、重点区域 3D 场景、处置资源布点建议、现场态势回写世界线。

## 34.4 系统输出

```text
现场状态摘要
空间态势图
现场证据片段
人流/车流/阻塞点
可用处置点位
风险区域边界
对 Live Incident 的事实修正
```

## 34.5 进入世界线与 Agent Council

无人机 / 3D 结果不直接生成结论，而是作为事实注入：

```text
现场观察 → 更新 Live Incident → 修正支点 → 更新世界线节点 → 带入 Agent Council
```

# 35. 更新后的关键产品原则

1. 系统有两条主链路：事前推演与事中快响。
2. 线上数据主要用于发现苗头、判断热度、识别误读与传播结构。
3. 全网快搜是事中处理的核心入口，不依赖系统此前是否识别到该事件。
4. 同城传播分析必须做圈层级分析，不能做个人画像。
5. 热度、事实可信度、成线风险必须分开显示。
6. Agent 消费结构化态势对象，不直接处理无约束原始数据流。
7. 无人机和 3D 建模是现场事实增强层，不是替代线上数据的主能力。
8. 数据源实时性能只能承诺系统内部处理时效，不能承诺全网绝对实时。
9. 页面流程指引要全局一致，不应只出现在某个页面。
10. 所有高风险判断必须保留“系统建议 → 人工确认”的机制。
