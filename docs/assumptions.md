# CollectiveEventTwin Assumptions

## 2026-05-09 S1 foundation execution

- S1 local/test bootstrap identity is `admin` / `admin12345` with role `system_admin`. This seed is created idempotently by the FastAPI foundation service only when the S1 auth path is exercised. Production credentials must be provisioned outside code and are not stored in migrations.
- New S1 endpoints under `/api/v1/auth/*`, `/api/v1/users`, `/api/v1/roles`, `/api/v1/audit-logs`, `/api/v1/reviews*`, `/api/v1/review-templates`, and `/api/v1/ops/*` use the v1.0 envelope shape. Existing P0 endpoints keep their legacy bare JSON shape until their page contracts are migrated, so current closed-loop pages and tests remain stable.
- S1 Review results are persisted in both `reviews` and append-only `review_results`; waiver and result changes also write `audit_logs`. This prevents review outcomes from existing only as frontend state or transient payloads.
- The P0 baseline Alembic migration keeps PostgreSQL vector support for production, but its JSON/vector defaults are guarded by dialect so local SQLite migration smoke tests can run without external PostgreSQL.

## 2026-05-09 S2 data-source governance execution

- Xi'an first-phase social-issue samples are generated as synthetic product data only when external data credentials are unavailable. They still pass through DataSource, CollectionJob, CollectionRun, RawRecord, RawRecordPayload, MediaAsset, MediaProcessingRun, LineageEdge, and AuditLog records, and every downstream object carries `synthetic=true` or `is_synthetic=true`.
- File, public-web, and media import endpoints can accept explicitly synthetic payloads for local delivery validation when no external data/key is available. They still create import runs, collection runs, raw records, payloads, media records, lineage, health counters, and audit logs.
- Official API sources without `api_key_ref` are allowed to be created only as blocked sources. Import attempts return a failed import run with `official_api_key_missing`, write error queue and retry queue records, and do not create raw records.
- Public-web URL validation and crawl-policy drafts are persisted in `data_sources.policy`, append `source_policies`, update `source_health`, and write `audit_logs`. Source configuration publication now writes append-only `data_source_versions` rows from AT-049 onward; no frontend-only crawl or version state is accepted.
- Official API base configuration, auth `secret_ref`, connection-test metadata, and pagination dry-run results are persisted in `data_sources.policy`, append `source_policies`, update `source_health`, and write `audit_logs`. `http://` official API base URLs are refused; `synthetic://` official API adapters are allowed only when explicitly persisted as synthetic and still run through the same API/DB/audit path.
- RSS feed URL and inspection metadata are persisted in `data_sources.policy`, append `source_policies`, update `source_health`, and write `audit_logs`. Without external RSS access, `synthetic://.../rss-*` adapters are accepted only as explicitly synthetic feed sources; non-RSS and empty synthetic feeds still return production 422 errors and cannot be treated as successful product data.
- File-upload source configuration stores allowlisted file extensions, import schema, and max file size in `data_sources.policy`, appends `source_policies`, creates `source_health`, and writes `audit_logs`. The actual file object upload and file-run binding remain scoped to AT-071/AT-072, so AT-043 only freezes source configuration and forbidden-type rejection.
- AT-071 file upload receipt uses a local object store path (`WORLDLINE_OBJECT_STORE_PATH`, default `artifacts/object-store`) when no external object-storage service is configured. The uploaded bytes are written to that object-store path, while PostgreSQL `file_objects` stores the storage key, object URI, checksum, scan status, policy refs, source refs, synthetic flag, lineage, and audit metadata. This is accepted as the local production-equivalent storage adapter for development; external S3/OSS-compatible storage can replace the adapter without changing the `/api/v1/uploads` or `file_objects` contract.
- Webhook sources generate a one-time `webhook_secret_once` only in the create response; PostgreSQL stores `secret_hash`, `secret_ref`, source key, endpoint, signature version, delivery window, and replay ledger in `data_sources.policy`. Local synthetic validation uses an in-process secret cache as the stand-in for a production secret manager; if a process restart loses that cache, webhook signing secrets must be reissued or resolved from the deployment secret manager before accepting new payloads.
- Manual source configuration stores the manual entry schema in `data_sources.policy`, appends `source_policies`, creates `source_health`, and writes `audit_logs`. AT-046 freezes source creation and permission gating only; manual raw-record entry is implemented under AT-074.
- DB import and object-storage source configuration stores connection/bucket metadata plus `secret_ref` only; plaintext passwords, DSNs, access keys, and secret keys are rejected before persistence. Synthetic DB connection tests and object-storage list probes are production service paths backed by `data_sources.policy`, `source_policies`, `source_health`, and `audit_logs`; real external adapters remain behind deployment secret manager configuration.
- Data source version publishing requires a successful persisted connection test before `POST /api/v1/data-sources/{id}/versions/publish` can create a `data_source_versions` row. `data_sources.policy.published_version`, `collection_jobs.payload`, and `collection_runs.payload` carry the published version id, version number, and config hash so new collection work is traceable to the active source configuration.
- Data source version rollback is append-only: `POST /api/v1/data-sources/{id}/versions/{version}/rollback` does not delete or mutate historical snapshots. It creates a new published rollback version from the target snapshot, records from/to version ids in payload/audit/source policy, and leaves existing collection jobs/runs pinned to the version they were created with; only newly created collection jobs adopt the rollback version.
- Data source disable/enable uses `PATCH /api/v1/data-sources/{id}/status` with explicit reason. Disabled sources update `data_sources.status`, `data_sources.policy.operational_state`, `source_policies`, `source_health`, and audit logs; they reject new collection jobs and new collection runs with `DATA_SOURCE_DISABLED` while leaving already persisted/running collection runs untouched.
- Data source health detail uses persisted `collection_runs` plus `source_health`: no collection history returns `status=unknown`; successful runs update `last_success` and health counters; disabled-source run attempts still return `409 DATA_SOURCE_DISABLED` but also persist a failed `collection_runs` record, `collection_run_events`, `source_health`, error queue, and audit evidence so `last_failure` and `error_rate` are traceable.
- Data source compliance labels are stored in `data_sources.policy.compliance` and summarized in `policy.collection_controls`. A data source version cannot be published until a successful connection test and compliance-ready authorization scope, authorization basis, retention period, and data classification are persisted; missing authorization basis returns `DATA_SOURCE_COMPLIANCE_BASIS_REQUIRED` instead of silently publishing.
- S2 adapter registration is an in-process startup registry for the first production source families (`public_web`, `official_api`, `rss`, `file_upload`, `manual`, `db_import`, `object_storage`). AT-054 freezes registry presence and required handler validation only; real adapter fetch/parse/store workflows are implemented in later collection-run slices.
- Explicit one-time collection jobs (`schedule=once`) require a published data-source version and persist `query`, `window`, `job_kind=once`, and version pin metadata in `collection_jobs.payload`. Legacy collection-job calls without `schedule` remain compatible for already frozen tests and are not treated as AT-056 production one-time jobs.
- Explicit cron collection jobs (`schedule=cron:<five-field-expression>`) require a published active data-source version and persisted `query` and `window` objects. AT-057 registers the schedule by writing `payload.job_kind=cron`, `cron_expression`, and `scheduler_registration` metadata on `collection_jobs`; the later collection-run scheduler slice can consume this durable registration without treating frontend state as business truth.
- Collection job creator filtering uses `collection_jobs.created_by_id` for user-created jobs. System-created jobs such as webhook deliveries can keep `created_by_id=null` and remain tenant-scoped; the frontend displays those as system-created rather than inferring business state locally.
- Manual collection-run starts create `pending` `collection_runs` and a durable `pending` `workflow_runs` ledger entry named `CollectSourceRunWorkflow`; actual adapter execution/completion is reserved for the worker/activity slice. Because current `workflow_runs.case_id` is non-null, S2 collection workflow runs use the system case anchor `CASE-S2-COLLECTION` until the workflow ledger is generalized beyond case-scoped runs.
- Collection job pause uses `collection_jobs.status=paused` plus `collection_jobs.payload.pause` as the durable scheduler-control record. It preserves existing `pending`, `running`, and `retrying` runs, writes `collection_run_events` for active runs without interrupting them, and blocks new manual/scheduled starts with `COLLECTION_JOB_PAUSED` until the resume slice changes the persisted job state.
- Collection job resume maps the plan wording `scheduled` to current runtime `collection_jobs.status=active`; the scheduler-ready state is stored in `collection_jobs.payload.resume.scheduler_state` as `scheduled` for scheduled jobs and `manual_ready` for legacy manual jobs. Resume is blocked while the linked data source is disabled or policy-blocked.
- Collection run cancel uses a persisted local worker-stop stand-in until an external worker controller is attached. The API writes `cancelling` request metadata, `cancel_requested` and `cancel_completed` events for the collection run and linked workflow run, then returns the durable terminal state `canceled`; completed, failed, or already terminal runs return `COLLECTION_RUN_TERMINAL` instead of silently ignoring cancellation.
- Collection run retry is an enqueue operation, not a fake successful collection. Retrying is allowed only from `failed` or `canceled` runs, creates a new `pending` collection run plus pending `CollectSourceRunWorkflow`, inherits the original input snapshot, and writes no raw records until the worker/activity slice actually processes the retry.
- Local browser verification must point Vite at the current workspace API with `VITE_API_BASE_URL=http://127.0.0.1:18082`; the Docker dev proxy on `localhost:8080` may run an older image and is not accepted as evidence for current code.
- Local API performance probes use direct loopback connections (`trust_env=False`) so workstation HTTP proxy settings cannot turn `127.0.0.1` calls into proxy-mediated 502 responses.
- DB import cursor state remains persisted in `data_sources.policy.db_import_cursor` for AT-083 rather than a new cursor table. The canonical cursor-state service exposes that persisted checkpoint, writes read audit evidence, and treats `last_db_import_scan` as the success-only checkpoint; failed scan ledgers must not advance the cursor.
- Uploaded file bytes, including PDFs, are stored only through the configured object-store path and are not persisted or audited as reversible inline base64 previews. `file_objects.payload.content_preview` may keep redacted byte-size/checksum metadata, while `content_preview_base64` remains null and `content_preview_redacted=true`.
- PDF text parsing uses `pypdf` as the primary parser engine and records `parser_engine` on run/file/page outputs. Pages without extractable text are persisted as `ocr_required` with `PDF_OCR_REQUIRED`; true OCR remains a later processor and cannot be silently treated as extracted text.
- DOCX text parsing uses deterministic OOXML ZIP/XML parsing for paragraph text and table-cell text. Encrypted or legacy Office container bytes fail explicitly as `DOCX_ENCRYPTED_UNSUPPORTED`; layout pagination, comments, footnotes, tracked changes, embedded images, and OCR are later document-processing concerns and cannot be silently converted into report facts.
- RSS item parsing reads AT-070 raw RSS payloads as JSON and also accepts a single RSS/Atom `item`/`entry` XML payload. `title` and `link` are required for a parsed output; `guid` is optional and falls back to a deterministic `link:<sha256>` parser identity. Re-running the parser for an already persisted RSS item identity counts `duplicate_count` and does not write another normalization row.
- Manual record schema validation uses `validate_manual_record` as the persisted validator. Required `location` must come from the submitted record (`location`, payload location/address/district, payload city id, or an explicitly submitted `city_id`) rather than the source policy or DTO default; a valid manual record writes raw record, validator run, algorithm run, clean draft normalization row, lineage, and audit before the frontend can treat it as accepted.
- Text standardization uses `normalize_text` as the canonical S2 cleaner behind `/api/v1/normalization-runs`. It strips HTML tags, decodes entities, removes control characters, collapses whitespace, lowercases text for deterministic matching, keeps sensitive masking from `raw_record_payloads.masked_text`, and marks empty post-cleaning output as `invalid` instead of silently treating it as usable clean text.
- Datetime standardization uses `normalize_datetime` behind `/api/v1/normalization-runs/datetime`. It only treats source-provided payload fields or raw payload text as event-time evidence; default ingestion timestamps on `raw_records.occurred_at` are not promoted into business time facts. Unparseable records become `review_required` cleaner outputs rather than guessed timestamps.
- Location standardization uses `normalize_location` behind `/api/v1/normalization-runs/location`. Until a larger geocoder is introduced, it uses a deterministic Xi'an first-stage district/address dictionary over payload location fields and raw text. Multiple district matches are emitted as candidate outputs requiring review instead of guessing one location.
- Source trust assignment uses `assign_source_trust` behind `/api/v1/normalization-runs/source-trust`. Explicit trust comes from persisted `data_sources.policy.source_trust.score` or equivalent persisted source payload fields; missing trust config uses deterministic source-type defaults and writes `SOURCE_TRUST_DEFAULTED` warnings in the cleaner output instead of silently manufacturing confidence.
- Sensitive-field detection uses `detect_sensitive_fields` behind `/api/v1/detector-runs/sensitive-fields`. It detects phone, ID-card, email, minor-name, and class-reference signals from persisted raw payload text, but detector outputs expose only redacted previews, positions, lengths, and type metadata; raw sensitive values must remain available only through permissioned raw payload storage and are not returned by the detector result.
- Sensitive-field redaction uses `redact_sensitive_fields` behind `/api/v1/redaction-runs/sensitive-fields`. Default raw-record detail and export surfaces return `raw_record_payloads.masked_text`; original raw payload text is split to `/api/v1/raw-records/{id}/original`, requires `data_source:raw_original`, and writes an audit event that records ids/hashes only, not original content.
- Rule dedupe uses `dedupe_by_hash_and_external_id` behind `/api/v1/deduplication-runs`. Existing `RawRecordDedupGroup`, `LineageEdge`, `AlgorithmRun`, and `RawRecord.payload` are the durable AT-099 persistence surface; no new `raw_records.duplicate_of` column is added. Same-source external-identity or content-hash duplicates write `duplicate_of` into `RawRecord.payload`, while cross-source exact matches remain candidate summaries and must not mutate raw-record business state.
- Semantic similar dedupe uses `semantic_dedupe_records` behind `/api/v1/deduplication-runs/semantic`. Until an external embedding provider/key is configured, the product uses a deterministic local shingle embedding provider named `synthetic_deterministic_shingle`; this is a real persisted algorithm path, marked `synthetic_embedding=true`, and still writes `DeduplicationRun`, `RawRecordDedupGroup`, `AlgorithmRun`, `LineageEdge`, audit, and raw-record candidate state. Semantic results are candidate-only and must not write `duplicate_of`; manual confirmation is reserved for AT-101. Per-record embedding failures make the run `partial` and persist error refs rather than silently skipping records.
- Dedupe candidate manual decisions use `POST /api/v1/clean-records/{id}/dedupe-decision` and persist the decision in `RawRecordDedupGroup.payload.dedupe_decision`, `RawRecord.payload.dedupe_decision`, `LineageEdge`, and `AuditLog`. No new decision table is added for AT-101 because the current review material is candidate groups; a later clean-record workbench can promote this JSON decision ledger into a dedicated table if concurrency, reporting, or approval workflow needs require it. Confirmed candidates write `duplicate_of`; split candidates explicitly remove/avoid `duplicate_of`.
- Clean record list uses `GET /api/v1/clean-records` as a PostgreSQL-backed API view over `raw_records`, `raw_record_payloads`, latest `raw_record_normalizations`, `raw_record_quality_issues`, and persisted dedupe JSON state. For AT-110, `clean_record_id` equals `raw_record_id` so existing dedupe-decision, raw-detail, lineage, and audit surfaces remain compatible; a later dedicated `clean_records` table can be introduced only if edit/version workflow requires a separate identity. The list endpoint writes `clean_record.list` audit evidence, never returns original raw text or full `normalized_text`, and exposes `status_counts_scope=page` because `status_counts` is a current-page display helper rather than a full result-set aggregation.
- Clean record detail uses `GET /api/v1/clean-records/{id}` as a PostgreSQL-backed aggregate over the same raw-record identity. It returns default raw content only as `masked_text`, clean normalization detail with sensitive values masked, extraction summaries from persisted signal lineage, quality issues, lineage edges, and access metadata. Original raw text remains split to `GET /api/v1/raw-records/{id}/original` behind `data_source:raw_original`; no dedicated clean-detail table is added for AT-111.
- Clean record status updates use `PATCH /api/v1/clean-records/{id}/status` and persist the manual status ledger in `RawRecord.payload.clean_record_status` for AT-112 rather than introducing a dedicated table. Allowed manual statuses are `valid`, `invalid`, and `review_required`; `invalid` and `review_required` block downstream signal extraction, while `invalid` is rejected with `CLEAN_RECORD_REPORT_LOCKED` once the raw record is referenced by a report, report claim, or task. The mutation writes `clean_record.status_updated` audit evidence and the S2 console may only render this backend state, not frontend-only status.
- Clean record quality scoring uses `score_clean_record_quality` behind `POST /api/v1/data-quality-runs` for AT-113. It computes deterministic `completeness`, `freshness`, `trust`, and `overall` scores from persisted `raw_records`, `raw_record_payloads`, and `data_sources.policy.source_trust`, writes a `DataQualityRun`, `RawRecordQualityIssue` rows, an `AlgorithmRun`, `LineageEdge` evidence, audit actions, and the latest score ledger in `RawRecord.payload.score_clean_record_quality`. No new table is added for per-record scores until score history querying or concurrent review workflows require it; the clean-record list/detail surfaces read the current score from the raw-record JSON ledger.
- Data quality issue listing uses `GET /api/v1/data-quality/issues` over persisted `raw_record_quality_issues`. `raw_record_quality_issues.tenant_id` is the canonical tenant-scope field for AT-114 list performance and is backfilled from `data_quality_runs.tenant_id`; response rows keep data-quality-run, raw-record, source, score, and evidence references while returning only redacted payload previews. The migration adds the column as nullable for safe backfill on existing local data, while the model and all new writes require a tenant id.
- The AT-293 Channel registry is a backend capability/configuration registry, not product runtime data and not a frontend fixture. `GET /api/v1/collection-channels` derives its rows from the server-side channel definition and currently registered source adapters; channels with a source type but no dedicated channel adapter contract, such as `image_file`, `video_file`, `livestream`, `audio_file`, and `webhook`, must return `degraded` with explicit warnings until AT-294 through AT-301 add and validate channel-specific adapter/schema contracts.
- The AT-294 adapter contract treats `discover`, `fetch`, `parse`, and `normalize` as the canonical channel-adapter startup/service methods. Legacy helper names such as `validate_config` may remain as non-contract internals for compatibility, but `/api/v1/adapters/capabilities` and `/api/v1/collection-channels/adapter-contract` must expose the canonical four-method contract. AT-294 validates registered adapters and reports future channel-specific adapters that are not registered yet as degraded channel warnings rather than startup failures.
- The AT-295 `web_page` channel schema is a backend JSON Schema/UI-schema contract for configuring public-web source policy. It does not replace the real `public_web` collection chain: URL validation, crawl-policy persistence, robots handling, rate-limit state, link discovery, fetch, raw-record storage, lineage, and audit still use the existing FastAPI/PostgreSQL workflow surfaces. The schema accepts `https`, `http`, and explicitly marked `synthetic` seed URLs so no-key environments can exercise the full product pipeline without private scraping.
- The AT-296 `official_api` channel schema uses credential references only. `secret_ref` is the only accepted secret-bearing field; plaintext fields such as `api_key`, `secret`, `token`, and `password` are forbidden by the schema contract and by the existing FastAPI DTO extra-field rejection. Synthetic official API base URLs remain allowed so the auth, connection test, pagination, fetch, raw-record, lineage, retry/dead-letter, and audit paths can be exercised without a real external key.
- The AT-297 `document_file` channel schema derives its allowlist from the existing `file_upload` backend policy validator. The schema drives frontend file-type selection, but FastAPI remains the authority for allowed extensions, schema mapping, max file size, upload scan, `file_objects`, file-run import, raw-record persistence, lineage, and audit. Unsupported file types such as `exe`, `js`, `html`, `zip`, shell scripts, and command files must be blocked by backend policy even if a client bypasses the UI; synthetic uploads are explicitly labeled and still use the real upload/import path.
- The AT-298 `image_file` channel schema uses the existing `media` source type and `POST /api/v1/imports/media` path as the production ingestion surface. Image OCR and VLM/caption behavior is deterministic in no-key environments and persists to `media_processing_runs`; those outputs are candidate evidence only and cannot support report facts without later evidence review. Media source policy validation normalizes allowed image formats, OCR policy, VLM policy, redaction policy, and max size. If redaction is disabled, the source remains creatable for risk visibility but receives `IMAGE_REDACTION_DISABLED_RISK`; default display and media-processing outputs still mask sensitive text.
- The AT-299 `video_file` channel schema also uses the existing `media` source type and `POST /api/v1/imports/media` path, but video sources must declare `media_kind=video_file` and a `large_video_policy`. Without external ASR/OCR/VLM keys, video keyframes, ASR, OCR, and VLM captions use deterministic synthetic processors, persist to `media_processing_runs`, and remain candidate-only evidence. Backend media policy validation rejects missing large-video policy with `VIDEO_LARGE_POLICY_REQUIRED`, rejects unsupported video formats, normalizes keyframe/ASR/OCR/VLM/redaction/max-size policy, and masks sensitive transcript/title text before persistence.
- The AT-300 `livestream` channel schema uses the `live_segment` source type and the existing media import/persistence path for synthetic live segments. Livestream sources must declare `stream_url`, `stream_protocol`, segment policy, buffer policy, and `retention_policy`; missing retention policy returns `LIVE_RETENTION_POLICY_REQUIRED`. Without an external live ingest service, synthetic streams still create real live_segment sources, raw records, media assets, media processing runs, lineage, and audit logs, with segment/buffer/retention policy copied into `media_processing_runs.output` and sensitive transcript/title text masked.
- The AT-301 `audio_file` channel schema uses the existing `media` source type and `POST /api/v1/imports/media` path. Audio sources must declare `media_kind=audio_file`, allowed formats, ASR policy, segmentation policy, and language policy. Backend policy validation rejects unsupported languages with `AUDIO_LANGUAGE_UNSUPPORTED`, rejects unsupported audio formats, normalizes segmentation/language/redaction/max-size policy, and persists deterministic synthetic ASR plus segment metadata to `media_processing_runs` as candidate-only evidence with sensitive text masked.
- The AT-302 channel error-code map is service-level backend logic, not a new persistence table. `GET /api/v1/collection-channels/error-codes` exposes `map_channel_error_codes` from the backend channel registry, writes audit evidence on every read, and covers current S2 run/import validation, policy, retry, upstream, security, storage, and media-processing error codes. Unknown requested codes remain visible as `classification=unknown` with `CHANNEL_ERROR_CODE_UNMAPPED` warnings so run-detail UI never invents a silent frontend explanation.
- Cleaning run metrics use `GET /api/v1/cleaning-runs/{cleaning_run_id}/metrics` for AT-115, where `cleaning_run_id` is the persisted `collection_runs.id` for the source-clean-extract pipeline. There is no separate `CleaningRun` table in this release; parsed, cleaned, extracted, failed, quality, and consistency metrics are derived from PostgreSQL `collection_runs`, `raw_records`, `raw_record_payloads`, `raw_record_normalizations`, raw-to-signal lineage, `raw_record_quality_issues`, `metrics_snapshots`, and audit logs.

## 2026-05-09 S3A Xi'an City page execution

- The S3A City page uses `xian` as the canonical first-stage city id. City events are persisted derivatives of PostgreSQL `raw_records`; no frontend fixture or frontend-only event state is accepted as product data.
- City event fact display is limited to source/evidence references from `raw_records` and media lineage. Event creation does not assert factual truth; it records `evidence_refs`, `synthetic` flags, and lineage for later S4/S4B evidence review.
- City map mode, event filters, source filters, and topic creation are backend interactions. The frontend may keep transient control state for responsiveness, but each business-meaningful action writes or reads through `/api/v1/cities/xian/*`.

## 2026-05-09 S3B topic situation execution

- The S3B Topic Situation page is backed by PostgreSQL `topics`, `city_events`, and `raw_records`. If no topic exists for `xian`, the page creates one through `POST /api/v1/topics` from the highest available city event; if no city event exists, it first triggers the documented synthetic Xi'an source path and keeps all synthetic flags and lineage.
- Topic situation snapshots are deterministic derived records, not factual findings. Every candidate mainline, stance sample, spread path, and source breakdown keeps evidence references from upstream city events/raw records and writes an audit entry when the snapshot is generated.

## 2026-05-09 S4A signal workbench execution

- The S4A Data / Signal Workbench page is backed by PostgreSQL `topics`, `raw_records`, `signals`, `signal_packages`, `lineage_edges`, and `workflow_runs`. If a topic has no signals, the page may trigger `POST /api/v1/extraction-runs` to run the real deterministic extraction path; it does not render frontend mock signals.
- S4A extracted signals are candidate interpretations, not factual findings. Each signal stores `input_refs`, `evidence_refs`, `algorithm_version`, synthetic flags, confidence inputs, and blocked-claim guardrails for S4B evidence review.
- Signal package create/add/remove actions are persisted through backend APIs and audit logs. Frontend state is allowed only for transient rendering and query invalidation.

## 2026-05-09 S4B evidence and media execution

- Evidence candidates are generated from persisted S4A signals and upstream raw/media refs. They are stored as `evidence` plus `evidence_reviews`; status `candidate` is review material only and cannot be used as a report fact until the review status is confirmed.
- Without external OCR/ASR/CV keys, media processing uses deterministic synthetic processors that still write `media_processing_runs`, `evidence_media_links`, `lineage_edges`, and audit logs. Processor output is explicitly marked as non-factual until evidence review.
- Redaction defaults to masked display using the same policy path as S2 raw payload masking. Original or media-derived text remains referenced through persisted records and is not exposed as an unreviewed report fact.

## 2026-05-09 S5 mainline / World State / stakeholder execution

- S5 mainline, World State, case-graph, and stakeholder outputs use deterministic production code while no external LLM/Agent key is available. Outputs are persisted, versioned, evidence-referenced, audit-logged, and marked synthetic whenever upstream inputs are synthetic.
- The React mainline page must resolve the current mainline through `GET /api/v1/mainlines?topic_id=...`; it must not keep the selected production mainline only in frontend state.
- Existing no-token legacy P0 mainline endpoints remain bare-response compatible for historical tests. Token-authenticated S5 calls use v1.0 envelopes and the production persistence path.

## 2026-05-09 S6 worldline / Agent Profile / Council execution

- Without an external LLM key, S6 uses a deterministic synthetic LLM provider identified as `LLM-SYNTHETIC-DETERMINISTIC-V1`. It is a production code path, not frontend mock data: every call writes `llm_calls`, prompt/template references, input refs, output payloads, blocked claims, review records, and audit logs.
- Agent Profiles can only be generated from a persisted Worldline Run and a stakeholder that exists in PostgreSQL. Council creation is blocked until the Agent Profile has files persisted and an `agent_profile` third-party review has passed its gate.
- Council results can be applied back to the Worldline Run only after the result is persisted, schema-valid, evidence-linked, reviewed as `council_result`, and passed by the Review Gate. Unsupported or evidence-missing claims remain in `blocked_claims` and cannot become applied conclusions.

## 2026-05-09 S7A report / approval / export / task execution

- S7A reports can only be generated from a reviewed and applied Council Result. The report service resolves the related Worldline Run, World State, Mainline, Topic, and evidence refs from PostgreSQL and rejects missing or mismatched context.
- Every `report_claims` row must carry persisted evidence refs. Claim validation failure moves the report to `claim_validation_failed` and blocks review submission, publication, and export until fixed.
- Report export is a persisted product artifact in `report_exports`. Without an external document rendering service, the first production export formats are markdown/json stored in PostgreSQL payload with a content hash and synthetic/production watermark.
- Report-linked tasks inherit report evidence refs and write `task_events` plus audit logs on creation and status changes. Legacy no-token P0 task endpoints remain compatible only as historical API fallback.

## 2026-05-09 S7B retrospective / case library / config execution

- Retrospective memory can only be created from a published or exported report. Knowledge items preserve report, claim, task, and evidence refs and cannot enter the case library until the retrospective review gate passes.
- Case library entries are derived from approved retrospective knowledge. Applying a case suggestion creates a persisted `case_library_applications` row with conflict summary and audit log; it does not mutate downstream production facts silently.
- Configuration versions are separate from retrospective memory. A config version must pass regression against approved case library entries, then pass a `config_version` review gate before publication. Rollback updates release/config status and records impact scope without deleting history.
- Frontend page body implementation must follow the existing static design page first when a corresponding static page exists.
