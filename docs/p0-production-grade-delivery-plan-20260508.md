# CollectiveEventTwin P0 Production-Grade Delivery Plan

Date: 2026-05-08

Status: superseding delivery contract

Supersedes:

- `docs/p0-react-migration-development-plan-20260508.md` as the overall MVP plan.
- The deterministic-only backend interpretation in `docs/p0-tr1-architecture-data-api-agent-design-20260508.md`.

The React migration plan remains valid only as a frontend workstream after the production backend spine is in place.

## Delivery Bar

The target is a production-grade P0 system, not a local fixture demo.

Explicit requirements:

- Data access may start with authorized/imported sources, but the ingestion framework must be real.
- Algorithms, workflow orchestration, Agent runtime, and LLM execution must be real production code.
- Every meaningful page interaction must go through the backend and persist or query database state.
- Product runtime must not use mock data, static runtime fixtures, or frontend-only business state.
- All product data displayed in `apps/web` must come from PostgreSQL through `apps/api`.

Allowed only for automated tests:

- Test fixtures inside test setup.
- Fake LLM providers used by tests to make assertions deterministic.

Not allowed in product runtime:

- Pre-materialized demo bundles standing in for pipeline output.
- Frontend mock files or `worldline-observer-current/mock`.
- Static replica iframes as product pages.
- Deterministic `run_council` pretending to be an Agent/LLM result.
- Temporal workflows that only return status strings without executing business activities.

## Current Gap

Current code has useful infrastructure but not a production backend business spine:

- FastAPI, SQLAlchemy, Alembic, PostgreSQL, Temporal, worker, Redis, and React exist.
- Current seed path writes downstream objects directly from fixtures.
- Current Temporal workflows are placeholders that return strings.
- Current Agent Council is deterministic service code, not LLM-backed agent execution.
- `p0-agent-runtime` contains useful spike logic but is not integrated into the production API/worker.

Therefore, backend productionization becomes the critical path before broad frontend migration.

## Target Backend Architecture

### 1. Database-First Data Model

Add or harden tables so every object is persisted before display:

- `data_sources`: configured source definitions, authorization mode, retention, policy, health.
- `collection_runs`: ingestion/import run records with status, counters, errors, and actor.
- `raw_records`: original imported/collected records, source metadata, payload, hash, timestamps.
- Existing tables remain canonical for product objects: `source_records`, `signals`, `evidence`, `risk_factors`, `mainlines`, `world_states`, `worldline_nodes`, `council_sessions`, `reports`, `tasks`, `audit_logs`, `workflow_runs`.
- Add persisted UI/business state where needed: selected case/page filters, selected signal/mainline/node, saved query, review status, applied council result, report/export status.

Rules:

- Product page APIs query database state only.
- Raw input can be inserted through import/ingestion APIs, but downstream objects must be generated by services/workflows.
- Seed/dev commands may insert authorized raw records for local development only, then run the same ingestion pipeline as production.

### 2. Real Ingestion Framework

Implement connector interfaces even if only authorized/imported adapters ship in P0:

- `SourceAdapter`: validates config, fetches/imports records, normalizes metadata, emits raw records.
- P0 adapters: manual upload/import, authorized export import, public web sample importer.
- `SourcePolicyService`: enforces public/authorized source rules before downstream processing.
- `RawRecordNormalizer`: canonicalizes text, media metadata, timestamps, source URL, region hints, author/publication metadata, and sensitivity hints.
- `CollectionRunService`: records run state, error details, source counts, blocked counts, and audit entries.

Out of P0:

- Full real Douyin/Kuaishou/Weibo/WeChat/Xiaohongshu connector onboarding.
- Private, login-bypass, cookie-pool, captcha-bypass, or non-public collection.

### 3. Production Algorithms

Promote spike logic into production services and make outputs reproducible and testable:

- `SignalExtractionService`: groups raw records, computes heat, credibility, tags, narratives, region, time, related evidence.
- `EvidenceService`: classifies fact, needs-review fact, rumor, opinion, emotion, propagation, and sensitive content; stores masked display text.
- `RiskFactorEngine`: maps evidence/signals to risk factors using a versioned rule set and confidence scoring.
- `MainlineBuilder`: builds support points, trigger clusters, evidence gaps, involved subjects, and draft mainlines from confirmed/eligible data.
- `WorldStateBuilder`: generates world state from confirmed mainline inputs.
- `WorldlineEngine`: computes 24-72 hour branches, probabilities, risks, support point changes, evidence refs, and council recommendation.
- `ReportComposer`: generates draft report, compliance notes, task candidates, and trace links.

All algorithms must:

- Use typed inputs from database records.
- Persist outputs before returning them to the frontend.
- Store algorithm version and input object refs in payload/audit.
- Be covered by unit tests using database-backed fixtures.

### 4. Real Workflow Orchestration

Replace placeholder workflows with Temporal workflows plus activities:

- `IngestCaseWorkflow`
  - create collection run
  - execute adapter/import
  - apply source policy
  - persist raw records/source records
  - normalize raw records
  - write audit/workflow state

- `BuildMainlineWorkflow`
  - extract signals
  - create evidence
  - generate risk factors
  - build or update mainline draft
  - persist world state only after confirmation or explicit workflow step

- `GenerateWorldlineWorkflow`
  - read confirmed mainline/world state
  - run worldline engine
  - persist nodes and explanations

- `RunCouncilWorkflow`
  - build evidence-bounded agent context
  - call Agent/LLM runtime
  - validate schema and evidence refs
  - persist council session and blocked claims

- `GenerateReportWorkflow`
  - compose report from evidence, worldline, council result, tasks, and compliance state
  - enforce human confirmation gate for formal conclusions

Each workflow/activity must be idempotent by case/object/run id and must write `workflow_runs` and `audit_logs`.

### 5. Agent And LLM Runtime

Implement a real Agent/LLM subsystem:

- `LLMProvider` interface with timeout, retry, structured output, token/cost metadata, and error taxonomy.
- Production provider: OpenAI-compatible or equivalent configured through environment variables.
- Test-only provider: deterministic fake available only in tests.
- `AgentProfile`: role, allowed evidence scope, prompt template version, output schema, safety constraints.
- `AgentCouncilRunner`: runs fixed role set per scenario, composes bounded context, invokes LLM, validates schema, and stores outputs.
- `AgentGuardrailService`: blocks unsupported factual claims, missing evidence refs, personal blame without evidence, privacy violations, and overconfident conclusions.

Rules:

- Agent output is never a fact source.
- Every claim must link to evidence refs or be placed in `blocked_claims`.
- LLM output must pass schema validation before persistence.
- Failed/invalid LLM calls must produce visible workflow errors, not silent fallback conclusions.
- Human confirmation remains mandatory before high-risk formal conclusions enter the report.

### 6. Backend-Backed Frontend Interactions

Every meaningful product interaction in `apps/web` must call the backend:

- Page route load fetches page view from API.
- Filters/search/rank tabs query backend page APIs with parameters and can persist saved view state where relevant.
- Map layer mode, selected event, selected signal, selected node, selected council session, and selected report/task state are backed by API state or query params resolved by the backend.
- Evidence status changes, factor decisions, signal inclusion/exclusion, mainline confirmation, council run/apply, pressure tests, report confirmation, task updates, memory/library/config actions all mutate backend state.
- Frontend-only state is limited to ephemeral visual affordances such as hover, tooltip visibility, and panel open/close animations.

No frontend component may compute business truth that is not already returned by the API.

## Delivery Phases

### Phase A: Backend Spine And Data Model

Deliver:

- Alembic migration for data sources, collection runs, raw records, and persisted page/selection state as needed.
- Service boundaries for ingestion, algorithms, workflow activities, Agent/LLM, and reporting.
- Remove direct fixture materialization as the product path.

Exit:

- Raw authorized records can be inserted/imported into DB and processed into downstream objects by services.

### Phase B: Ingestion And Algorithm Pipeline

Deliver:

- P0 source adapters for manual upload/import, authorized export import, and public web sample import.
- Signal/evidence/factor/mainline/world state/worldline/report production services.
- Third non-campus scenario `CASE-MARKET-FOOD-001` processed through the pipeline, not prewritten as downstream fixture rows.

Exit:

- `CASE-CAMPUS-001`, `CASE-COMMUNITY-WATER-001`, and `CASE-MARKET-FOOD-001` all enter from DB raw records and complete the object chain.

### Phase C: Temporal Workflows

Deliver:

- Real Temporal activities for ingestion, processing, council, report, and task generation.
- Workflow idempotency, visible failure state, retry behavior, audit, and run records.

Exit:

- Starting workflows through API changes database state and can be observed in `workflow_runs` and audit.

### Phase D: Agent/LLM Production Runtime

Deliver:

- LLM provider interface and production provider.
- Agent profiles and prompt/context builder.
- Schema validation, evidence-bound guardrails, blocked claims, token/cost/error logging.
- Test-only fake provider for deterministic CI tests.

Exit:

- Agent Council results are produced by the runtime, persisted, schema-valid, evidence-linked, and auditable.

### Phase E: API-Backed Product UI

Deliver:

- Keep `city` as first frozen UI page, but rewire all business data/interactions to backend state.
- Migrate `risk`, `data`, `evidence`, `mainline`, `worldline`, `council`, `brief`, `memory`, `library`, and `config` only after backend endpoints exist.
- Remove or quarantine static replica runtime paths from product routing.

Exit:

- Product UI has no mock/static runtime dependency and no frontend-only business interaction.

### Phase F: TR4 / DCP Readiness

Deliver:

- Backend unit/integration tests for all services and workflows.
- Agent/LLM schema and guardrail tests.
- API tests for every page interaction.
- Playwright E2E for the full closed loop across all three cases.
- Docker smoke with API, worker, DB, Redis, Temporal, and web.
- Production readiness report with remaining non-P0 items.

Exit:

- System qualifies for DCP as a production-grade P0 candidate, excluding only real platform connector onboarding.

## Acceptance Criteria

P0 is not accepted until:

- No product page reads static mock data or frontend fixtures.
- No downstream business object is created only by fixture materialization in the product path.
- Every product interaction either queries or mutates backend state.
- Temporal workflows execute real activities.
- Agent Council uses real LLM runtime in product configuration.
- All LLM outputs are schema-valid, evidence-linked, auditable, and guarded.
- The three required scenarios complete the same database-backed business chain.
- Docker local stack can run the system end to end.

## Non-P0 / Later Hardening

These are still post-P0 unless explicitly re-scoped:

- Real social platform connector onboarding.
- Full RBAC and multi-tenant organization model.
- Production observability stack with dashboards and alerts.
- OpenSearch adapter.
- Full commercial deployment automation.
