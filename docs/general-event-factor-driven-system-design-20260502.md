# 通用事件因子驱动系统设计方案

日期：2026-05-02

## 1. 核心纠偏

校园暴力、学生跳楼、家属聚集只是一个例子，不能为每一种事件单独设计一套模板。

现实中的事件类型会非常多：

- 校园安全。
- 医疗纠纷。
- 工地事故。
- 食品安全。
- 文旅投诉。
- 房地产维权。
- 劳资纠纷。
- 交通事故。
- 老旧小区改造。
- 环境污染。
- 价格争议。
- 宠物、噪音、邻里冲突。
- 企业裁员。
- 商场踩踏或消防隐患。
- 政策执行争议。

如果每一类都做一个完整模板，会导致系统无法维护，也无法覆盖真实世界的组合型事件。

正确做法不是“事件模板穷举”，而是：

```text
通用事件骨架
+ 风险因子识别
+ 主体立场识别
+ 证据链管理
+ 线下/线上转化判断
+ 动态任务编排
= 可适配多场景的事件处置系统
```

## 2. 从“模板系统”转向“因子系统”

### 2.1 模板系统的问题

模板系统的隐含假设是：

```text
事件类型是稳定的；
每类事件有固定流程；
用户能一开始准确选择模板；
处置动作可以提前写死。
```

但真实情况是：

- 事件一开始类型不清。
- 一个事件经常横跨多个领域。
- 风险点会随着传播变化。
- 用户没有时间先选模板。
- 处置流程要根据事实、情绪、线下状态动态变化。

例如：

```text
“景区排队投诉”
可能演变成：
游客踩踏风险 + 价格争议 + 服务承载争议 + 官方回应不足。

“小区停水”
可能演变成：
民生保障 + 老旧管网 + 物业矛盾 + 线下聚集 + 政府部门责任。

“医院纠纷”
可能演变成：
医疗事故争议 + 家属聚集 + 现场冲突 + 网络爆料 + 医闹标签争议。
```

### 2.2 因子系统的核心

系统不强迫用户先选择“这是什么事件”，而是自动识别事件中出现了哪些风险因子。

事件可以被表示为：

```text
事件 = 基础事实 + 涉及主体 + 风险因子 + 证据状态 + 传播状态 + 线下状态 + 响应状态
```

每个风险因子都会影响：

- 风险等级。
- 需要看的页面模块。
- 需要生成的任务。
- 需要提醒的角色。
- 需要推演的世界线。
- 需要保护的信息。

## 3. 通用事件骨架

所有事件都共享一套底层结构。

### 3.1 事件基础信息

```text
发生了什么
在哪里发生
什么时候发生
涉及哪些主体
谁最先发布
当前热度如何
事实确认到哪一步
是否已有官方/机构回应
是否出现线下行动
```

### 3.2 七个通用面板

系统页面不按具体事件类型拆，而按所有事件都会遇到的判断问题拆。

| 面板 | 回答的问题 | 适用场景 |
| --- | --- | --- |
| 事实面板 | 哪些事实已确认，哪些待核验 | 全部事件 |
| 主体面板 | 谁被卷入，各方立场是什么 | 全部事件 |
| 证据面板 | 哪些内容支撑判断，哪些只是传言 | 全部事件 |
| 传播面板 | 热度是否上升，是否跨平台扩散 | 全部事件 |
| 情绪面板 | 公众为什么愤怒、恐惧、同情或不信任 | 全部事件 |
| 线下面板 | 网络情绪是否转化为现实行动 | 部分事件 |
| 响应面板 | 机构/官方/企业回应是否有效 | 全部事件 |

这七个面板是稳定的，事件类型只影响每个面板中的重点字段。

## 4. 风险因子库

风险因子是系统的核心配置单元。

### 4.1 高敏事实因子

| 因子 | 说明 | 影响 |
| --- | --- | --- |
| 死亡 | 出现人员死亡 | 风险直接升高，要求事实核验和审慎表述 |
| 重伤 | 出现严重伤害 | 触发现实处置和责任调查 |
| 未成年人 | 涉及学生、儿童、未成年当事人 | 默认隐私保护，禁止扩散身份 |
| 老人/弱势群体 | 涉及老人、残障、困难群体 | 情绪敏感度提高 |
| 食品/药品安全 | 涉及健康和公共安全 | 触发监管和溯源任务 |
| 公共交通/大型场所 | 涉及人群密集空间 | 关注次生风险和秩序 |
| 公共资源分配 | 涉及教育、医疗、住房、保障 | 容易转为公平叙事 |

### 4.2 责任归因因子

| 因子 | 说明 | 影响 |
| --- | --- | --- |
| 指向学校 | 学校、教师、教育管理被指责 | 触发教育治理视角 |
| 指向医院 | 医生、医院、医疗流程被指责 | 触发医疗合规与家属沟通视角 |
| 指向企业 | 企业服务、劳动关系、安全生产被指责 | 触发企业责任和监管视角 |
| 指向政府部门 | 执法、监管、政策执行被指责 | 触发公共治理风险 |
| 指向物业/社区 | 基层服务、物业管理被指责 | 触发民生协调风险 |
| 指向个人 | 具体自然人被曝光 | 触发隐私、网暴和事实核验 |

### 4.3 传播爆燃因子

| 因子 | 说明 | 影响 |
| --- | --- | --- |
| 短时评论暴增 | 评论量短时间快速增长 | 提升爆燃指数 |
| 现场视频出现 | 具有强情绪感染力的现场画面 | 提升传播风险 |
| 直播出现 | 事件实时化 | 提升现场风险和扩散速度 |
| KOL/媒体介入 | 大账号开始转发或解读 | 触发跨圈层传播 |
| 跨平台迁移 | 从同城到微博/新闻/论坛扩散 | 风险范围扩大 |
| 话题标签形成 | 出现稳定话题词 | 形成议题主线 |

### 4.4 线下行动因子

| 因子 | 说明 | 影响 |
| --- | --- | --- |
| 家属到场 | 家属前往机构/现场 | 触发沟通和现场秩序任务 |
| 群体聚集 | 多人到现场 | 触发现实风险面板 |
| 拉横幅/喊话 | 明确对抗性表达 | 触发现场稳控任务 |
| 围堵/冲突 | 秩序风险上升 | 触发高等级预警 |
| 集体投诉/联名 | 网络诉求转组织化行动 | 触发部门协同 |
| 线下打卡围观 | 公众围观转现场压力 | 关注次生风险 |

### 4.5 信任破裂因子

| 因子 | 说明 | 影响 |
| --- | --- | --- |
| 质疑隐瞒 | 出现“压事”“不公开”等表达 | 提升回应压力 |
| 质疑不作为 | 出现“早知道但没管” | 责任归因上升 |
| 质疑甩锅 | 认为机构推卸责任 | 回应可信度下降 |
| 质疑程序不公 | 认为调查方不独立 | 需要第三方/联合调查说明 |
| 质疑区别对待 | 涉及权力、身份、资源不平等 | 容易进入公平叙事 |

### 4.6 敏感信息因子

| 因子 | 说明 | 影响 |
| --- | --- | --- |
| 未成年人身份曝光 | 姓名、照片、班级、学校等扩散 | 自动打码和审计 |
| 个人隐私曝光 | 电话、住址、身份证等 | 阻断外发和提示合规 |
| 涉事人被网暴 | 明确个人受到集中攻击 | 触发保护和降噪任务 |
| 谣言快速传播 | 未核实信息高传播 | 触发辟谣核验任务 |

## 5. 事件类型只是“初始预设”，不是模板边界

系统可以保留事件类型，但事件类型只用于初始化权重和默认关注点。

| 事件类型 | 默认关注因子 |
| --- | --- |
| 校园类 | 未成年人、学校责任、家属到场、隐私保护、欺凌/管理争议 |
| 医疗类 | 伤亡、家属聚集、医疗责任、病历证据、医患冲突 |
| 文旅类 | 人流承载、价格争议、服务投诉、现场视频、节假日扩散 |
| 房地产类 | 资金安全、延期交付、业主聚集、合同争议、政府协调 |
| 劳资类 | 欠薪、裁员、集体维权、企业责任、监管介入 |
| 食品安全类 | 健康损害、检测证据、监管责任、品牌扩散、区域影响 |
| 交通类 | 伤亡、道路/车辆责任、现场秩序、官方通报、次生拥堵 |
| 社区民生类 | 停水停电、物业矛盾、老人儿童影响、基层响应 |

用户可以一开始不知道事件类型，系统也可以先以“未分类高敏事件”进入。

随后系统根据风险因子动态调整页面重点和任务。

## 6. 动态流程编排

流程不是按事件模板写死，而是由风险因子触发。

### 6.1 示例：出现“死亡”因子

系统自动增加：

- 死亡事实核验任务。
- 审慎表述提示。
- 高等级预警。
- 责任调查主线。
- 官方回应压力。

### 6.2 示例：出现“未成年人”因子

系统自动增加：

- 隐私保护。
- 身份信息打码。
- 外发材料脱敏。
- 涉事人网暴监测。
- 未成年人保护提示。

### 6.3 示例：出现“群体聚集”因子

系统自动增加：

- 线下风险面板。
- 现场人数录入。
- 现场联络任务。
- 分流沟通建议。
- 现场视频监测。

### 6.4 示例：出现“质疑隐瞒”因子

系统自动增加：

- 回应可信度评估。
- 信任真空倒计时。
- 核心质疑清单。
- 下一次信息更新时间建议。

## 7. 页面设计从“事件模板页”改为“模块自适应工作台”

### 7.1 固定底座

所有事件工作台都有：

```text
顶部状态条
左侧信号时间轴
中央事件判断区
右侧任务与缺口
底部推演与回应
```

### 7.2 动态模块

根据风险因子自动出现或置顶：

| 触发因子 | 置顶模块 |
| --- | --- |
| 死亡/重伤 | 事实核验、责任主线、回应边界 |
| 未成年人 | 隐私保护、敏感信息拦截 |
| 群体聚集 | 线下风险、现场任务 |
| 直播/现场视频 | 传播爆燃、证据链 |
| 质疑隐瞒 | 回应可信度、信任真空 |
| 多方对立 | 立场图谱、叙事战场 |
| 官方已回应 | 回应效果评估 |

用户看到的不是固定模板，而是系统根据事件状态自动调整的工作台。

## 8. 用户交互逻辑

### 8.1 用户不选模板，用户确认因子

错误设计：

```text
请选择事件模板：
校园欺凌 / 医疗纠纷 / 食品安全 / 劳资纠纷
```

正确设计：

```text
系统识别到以下风险因子：
- 未成年人
- 学校
- 死亡待确认
- 家属到场
- 质疑不作为

是否确认并创建高敏事件？
```

### 8.2 系统给结构，不给结论绑架

系统可以提示：

```text
当前存在“校园欺凌导致死亡”的叙事，但证据状态为待核验。
```

而不是直接写：

```text
这是校园欺凌事件。
```

### 8.3 用户可以修正因子

用户应能：

- 删除误识别因子。
- 添加遗漏因子。
- 调整可信度。
- 合并重复事件。
- 将传言降级为待核验。
- 将事实升级为已确认。

所有修正进入审计日志。

## 10. MVP 实现策略

第一版不做无限复杂的智能系统，而是做可配置因子引擎。

### 10.1 MVP 必做

- 通用事件工作台。
- 风险因子识别与人工确认。
- 七个通用面板。
- 动态模块置顶。
- 因子触发任务。
- 证据链管理。
- 会商简报生成。

### 10.2 MVP 因子数量

建议第一版只做 30-50 个高价值因子，而不是上百个。

优先：

- 死亡。
- 重伤。
- 未成年人。
- 老人/弱势群体。
- 学校。
- 医院。
- 企业。
- 政府部门。
- 家属到场。
- 群体聚集。
- 直播。
- 现场视频。
- KOL介入。
- 跨平台传播。
- 质疑隐瞒。
- 质疑不作为。
- 官方回应。
- 隐私曝光。
- 网暴。
- 集体投诉。

### 10.3 后续扩展

第二阶段再增加行业领域配置：

- 教育。
- 医疗。
- 文旅。
- 房地产。
- 劳资。
- 食品安全。
- 交通。
- 社区民生。

领域配置只改变权重和默认字段，不另起一套系统。

## 11. 最终产品形态

系统不再是：

```text
校园暴力模板
医疗纠纷模板
文旅投诉模板
劳资纠纷模板
```

而是：

```text
通用事件工作台
+ 风险因子库
+ 动态任务编排
+ 证据链
+ 立场与叙事
+ 世界线推演
+ 会商与处置
```

一句话：

> 事件类型只是入口，风险因子才是系统真正的操作单元。
