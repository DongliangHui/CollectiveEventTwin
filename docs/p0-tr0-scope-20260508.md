# CollectiveEventTwin P0 TR0 范围文档

日期：2026-05-08

状态：TR0 scope accepted，用户已确认方案 A，进入 P0 PRD / stories / TR1

适用项目：CollectiveEventTwin / Worldline Observer

## 1. TR0 结论

P0 应定义为：

> 基于授权或测试数据的网络风险世界线闭环系统。它把网络信号转成可复核的风险因子、证据链、主线、World State、世界线分支、Agent Council 研判、决策简报和处置任务，但不替代事实调查、线下执法、官方决策或真实民意。

P0 不应定义为：

- 全量城市社会风险平台。
- 全网实时舆情采集系统。
- 多智能体自动社会仿真系统。
- 静态页面继续堆功能。
- 未经授权的平台爬虫或数据采集项目。

TR0 决策：

采用“端到端闭环优先”的 P0 方案。先以 `CASE-CAMPUS-001 / campus-death-high-intensity` 为黄金验收场景，并增加至少一个非校园轻量 smoke fixture 验证对象模型不是硬编码校园模板。真实平台连接器、全网快搜、生产级实时流、无人机/3D、多租户和复杂案例库进入 P1 或更后。

TR0 状态：

已通过进入 P0 PRD 与 TR1 设计。实现仍被阻塞，直到 P0 PRD、stories/AC、screen inventory 和 TR1 设计完成。

## 3. P0 用户可见结果

P0 完成后，用户应能在本地或测试环境中完成一条真实业务形态的闭环：

1. 从高敏事件或网络信号进入系统。
2. 查看信号来源、可信度、风险因子、地理/传播态势和证据缺口。
3. 区分事实、传言、观点、情绪和传播结构。
4. 人工确认或修正风险因子，必要时带缺口进入主线建模。
5. 生成并确认主线与 World State 输入包。
6. 运行 24-72 小时世界线推演，查看关键分支、概率和支点变化。
7. 对关键节点启动 Agent Council，获得结构化多主体反应和概率回写。
8. 生成决策简报、核验任务、回应准备任务和后续监测任务。
9. 在全过程保留来源、证据、人工确认、模型输出和导出审计。

P0 的成功标准不是“接入多少真实平台”，而是证明：

```text
授权/测试数据
-> Signal
-> RiskFactor
-> Evidence
-> Mainline
-> WorldState
-> WorldlineNode
-> CouncilResult
-> Report
-> Task
-> Audit
```

这条链路可解释、可复核、可测试、可扩展。

## 4. P0 范围

### 4.1 P0 必须包含

| 范围 | P0 要求 |
| --- | --- |
| 来源策略 | 只允许 `public_web`、`official_api`、`authorized_export`、`manual_upload`、`third_party_contract`、`test_fixture` 进入处理链。 |
| 数据对象 | 定义统一对象：SourceRecord / RawData / Signal / RiskFactor / Evidence / Event / Mainline / WorldState / WorldlineNode / CouncilSession / Report / Task / AuditLog。 |
| 黄金场景 | 以校园死亡高烈度事件为完整验收场景：死亡事实、家属聚集、短视频扩散、回应可信度、未成年人隐私。 |
| 泛化验证 | 至少一个非校园 smoke fixture，用于验证因子系统不是单案例模板。 |
| 因子引擎 v0 | 支持高价值因子识别和人工确认。优先覆盖死亡、未成年人、家属到场、现场视频、质疑隐瞒、质疑不作为、线下聚集、官方回应、隐私曝光、网暴、证据缺口、跨平台传播。 |
| 证据链 | 每条高风险判断必须能追溯到来源、摘录、可信度、状态和人工处理记录。 |
| 主线建模 | 支持从信号和因子生成主线草稿，展示支点、诱因、证据缺口，并要求人工确认。 |
| 世界线推演 | 支持基于 World State 生成有限分支、节点风险、概率、支点变化和假设注入入口。 |
| Agent Council | Agent 只能消费结构化 World State / node context，输出结构化反应、支点 delta、分支概率 delta、依据和不确定性。 |
| 决策简报 | 生成最终研判、关键证据、证据缺口、建议动作、任务清单、后续监测点和合规提示。 |
| 当前静态页契约 | 当前 `worldline-observer-current` 黄金路径作为 P0 行为和数据契约参考，不把静态 HTML 本身当生产架构。 |
| 本地可验证 | 在本地测试数据下完成完整链路，避免先依赖外部平台。 |
| 合规护栏 | 默认脱敏未成年人、个人身份、头像、班级、住址等敏感信息；禁止个人画像和个人处置建议；保留审计。 |

### 4.2 P0 可包含但需控制

| 范围 | 控制方式 |
| --- | --- |
| 数据源中心 | P0 可先做来源注册表、状态、授权模式和健康状态，不做完整商业化数据治理后台。 |
| 案例库 | P0 可保存一次简化复盘记录，不做完整案例检索、偏差库和 Agent 校准库。 |
| 地图态势 | P0 可使用静态地理点、热区和传播路径，不做生产 GIS 或实时空间计算。 |
| LLM 解释 | P0 可调用 LLM 生成解释和结构化研判，但必须通过 schema validator、证据边界和人工确认。 |
| 报告导出 | P0 可先支持 markdown/json/html 级简报，不强制 PDF/PPTX/DOCX 全套导出。 |

### 4.3 P0 明确不做

| 不做项 | 原因 |
| --- | --- |
| 绕过权限、反爬、登录态、cookie 池、验证码的平台抓取 | 合规风险不可接受。 |
| 非公开群聊、私密账号、个人关系链或个人位置追踪 | 违反产品边界和隐私原则。 |
| 把线上热度直接判定为现实事实或线下风险 | 必须事实/情绪/传播三分离。 |
| 自动发布官方回应或直接指挥线下执法 | 系统只做研判、建议和任务辅助。 |
| 把 Agent 输出当真实民意、调查结论或最终裁决 | Agent 是压力测试工具，不是事实来源。 |
| 生产级全网快搜和实时流处理 | 外部源授权、时效和稳定性不可在 P0 直接承诺。 |
| 无人机、3D、视频流实时分析 | 属于现场态势增强层，P0 不需要。 |
| 多租户、复杂权限组织、商业化配置市场 | 不是第一条闭环的必要条件。 |
| 全量 20+ 页面一次性交付 | P0 先锁核心闭环页面和对象。 |

## 6. P0 验收边界

### 6.1 端到端验收

1. 从 P0 入口进入 `CASE-CAMPUS-001`，能完成风险发现到简报任务的完整链路。
2. 页面或 API 状态不能出现死链、空状态、ID 丢失或对象断裂。
3. 数据页加入信号后，主线页能识别同一 `mainlineId`。
4. 主线确认后，世界线页能使用同一 `worldStateId`。
5. Agent Council 注入后，世界线节点和简报能展示概率/支点变化。
6. 简报能生成任务，任务能回写状态和触发来源。

### 6.2 专业验收

1. 所有高风险判断展示证据来源、可信度、更新时间和人工确认状态。
2. 事实、传言、观点、情绪、传播结构不可混写为同一结论。
3. 因子变更会影响任务建议、主线支点或推演输入，并留下审计。
4. Agent 输出必须结构化，不能新增无证据事实。
5. 未成年人和个人敏感信息默认脱敏。
6. 未经人工确认的高风险判断不能进入正式简报结论区。

### 6.3 技术验收

P0 技术目标参考全量 PRD，但按本地闭环收敛：

| 能力 | P0 目标 |
| --- | ---: |
| 数据 / 信号检索 | < 3s |
| 相似或相关信号推荐 | < 5s |
| 主线加载 | < 3s |
| 世界线推演 | < 180s |
| Agent Council | < 180s |
| 简报生成 | < 90s |
| 人工上传或 test fixture 入库 | < 10s |

### 6.4 TR1 前必须补齐的验收材料

## 7. 四视角 TR0 评审

### 7.1 产品视角

结论：P0 产品价值成立，但必须收敛到“闭环”而不是“全量平台”。

主要发现：

- 全量 PRD 覆盖数据源、检索、主线、推演、Agent、事中快响、案例库、无人机/3D、模型运营和审计，不能直接作为 P0 实现范围。
- 当前静态页已经证明了黄金路径，但静态页是行为和信息架构样本，不是生产实现本身。
- 高烈度校园事件适合作为 P0 黄金场景，因为它同时覆盖事实未清、情绪爆燃、多方立场、线下聚集、回应可信度和未成年人隐私。
- P0 必须避免变成单案例脚本，所以需要一个非校园 smoke fixture 验证因子系统泛化。

产品决策：

- P0 采用“授权/测试数据端到端闭环”定义。
- 当前静态黄金路径作为行为契约，不立即重做所有页面。

### 7.2 架构视角

结论：现有 `p0-agent-runtime` 是有价值的 tracer-bullet，但不能直接提升为生产架构。TR1 前必须先锁对象模型、API、状态流转和合规 ingestion gate。

主要发现：

- 当前 mock API 已包含 `Signal`、`Mainline`、`WorldState`、`WorldlineNode`、`CouncilResult`、`Report`、map feature 等对象，适合转成 API contract。
- `p0-agent-runtime` 已有 source registry、policy filter、signal/map/mainline/worldline/council/report output，适合作为 spike 证据，但生产应避免“写 JSON 再手工复制到前端 fixtures”的集成方式。
- Agent 不能直接消费无约束原始流，应消费结构化态势对象和证据引用。
- P0 技术栈可收敛为 FastAPI + PostgreSQL/JSONB + Redis/queue + LLM Gateway + validator。OpenSearch、Kafka、Flink、Graph DB、Temporal 等生产增强不应默认进入 P0。

架构 blocker：

- 缺少正式 object schema 和 API contract。
- 缺少状态机定义：signal / evidence / factor / mainline / node / council / report / task。
- 缺少 LLM 输出校验、重试、降级和审计设计。
- 缺少 ingestion policy gate 的正式设计。

### 7.3 QA 视角

结论：现有 demo 验收只证明“页面能走通”，不足以证明 P0 可交付。TR1 前需要把验收拆成行为、数据契约、合规和回归测试。

主要发现：

- 黄金路径验收应继续保留：入口 -> 数据页 -> 主线 -> 世界线 -> Agent Council -> 简报。
- 需要新增负向验收：未授权 source 被拒绝、含未成年人信息默认脱敏、无证据 Agent 输出被拦截、未人工确认结论不能进正式简报。
- 需要 object contract tests，防止静态 mock、runtime output 和 API schema 逐渐分叉。
- 需要至少两类 fixture：完整校园黄金场景和非校园最小泛化场景。

QA blocker：

- 缺少 P0 stories 和 acceptance criteria。
- 缺少测试数据分层：golden / smoke / negative / compliance。
- 缺少报告、任务、审计的可重复验证方式。

### 7.4 数据 / 合规视角

结论：P0 可以做，但必须把数据来源和隐私边界作为一级产品能力，而不是实现细节。

主要发现：

- 公开网络信号适合 P0 早期价值，但不能把网络热度当事实真相。
- 允许来源必须显式标记授权模式、采集边界、保留策略和可信度。
- 校园死亡场景涉及未成年人、死亡事实、疑似欺凌和网暴，必须默认脱敏、最小化保留、证据索引优先。
- 系统不能做个人画像、个人追踪、非公开群聊采集、个人处置建议或舆论操控建议。

数据 / 合规 blocker：

- 缺少 source authorization matrix。
- 缺少敏感信息分类和默认脱敏策略。
- 缺少 evidence retention policy。
- 缺少 audit log 需求细化。
- 真实平台 connector 不能进入 P0，除非先完成授权和合规审查。

## 9. 可选方案

### 方案 A：授权/测试数据端到端闭环，推荐

范围：

- 使用测试数据、manual upload 或授权样例。
- 完成 Signal -> Evidence -> Factor -> Mainline -> Worldline -> Council -> Brief -> Task。
- 当前静态页作为行为契约和验收参照。
- 不做真实平台 connector。

优点：

- 合规风险低。
- 可快速形成工程闭环。
- 有利于先锁对象、API、Agent 和测试。
- 符合“数据源集成最后”的项目记忆。

代价：

- P0 对外实时数据价值有限。
- 需要清楚标注这是产品闭环 P0，不是全网数据能力 P0。

### 方案 B：有限公开源 connector P0

范围：

- 在方案 A 基础上接入少量公开新闻、政务公告或官方 API。
- 社媒/短视频仍只走授权样例或人工导入。

优点：

- 更接近真实产品价值。
- 可以验证 source registry 和 ingestion policy。

代价：

- 授权、频控、留存、版权、稳定性会拖慢 P0。
- 容易把 TR1 前架构工作变成 connector 调试。

### 方案 C：继续静态页面和 demo polish，不推荐

范围：

- 继续完善 `worldline-observer-current` 页面。

优点：

- 演示效果快。

代价：

- 不解决真实对象模型、API、合规、审计和可测试性。
- 容易积累不可落地的 UI 债务。

### 方案 D：按全量 PRD 开发，不推荐

范围：

- 同时做数据源、实时流、主线、世界线、Agent、事中快响、案例库、无人机/3D、审计、多租户。

优点：

- 愿景完整。

代价：

- P0 范围不可控。
- 架构、QA、合规风险同时放大。
- 难以形成可验收交付。

## 10. 建议决策

用户已确认以下决策：

1. P0 采用方案 A：授权/测试数据端到端闭环。
2. P0 黄金场景锁定 `CASE-CAMPUS-001 / campus-death-high-intensity`。
3. P0 增加一个非校园 smoke fixture，防止系统硬编码成校园事件模板。
4. 当前 `worldline-observer-current` 作为行为和数据契约参考，不作为继续堆功能的生产实现。
5. P0 暂不接真实 Douyin、Kuaishou、Weibo、WeChat、Xiaohongshu、论坛等 connector。
6. P0 的核心验收以对象链路、证据追溯、人工确认、Agent 结构化输出、简报任务闭环和合规护栏为主。
7. TR0 后下一步不是编码，而是产出 P0 PRD、stories/AC、screen inventory、TR1 架构/数据/API/Agent 设计和 delivery plan。

| 架构可行性 | 可行，但 TR1 前不得继续实现。 |
| QA 可验收性 | 可行，但需 stories/AC 和 test plan。 |
| 数据/合规 | 可行，仅限授权/测试数据；真实 connector 阻塞。 |

Gate 结论：

```text
TR0 pass to P0 PRD/TR1.
Implementation blocked until P0 PRD, stories/AC, screen inventory, and TR1 artifacts exist.
```

## 12. TR0 后续产物顺序

1. P0 PRD：只覆盖本文收敛后的 P0。
2. Stories / acceptance criteria：按页面、对象、合规和 Agent 输出拆分。
3. Screen inventory：锁定 P0 页面、入口、状态和跳转。
4. TR1 architecture：API、schema、storage、queue、LLM gateway、validator、audit。
5. Data / compliance design：source matrix、retention、masking、audit、人工确认策略。
6. Agent design：Agent 输入对象、输出 schema、prompt boundaries、fallback。
7. Delivery plan：垂直切片、测试数据、验证命令和受控 agent dispatch。
